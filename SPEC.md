# 系統規格文件
## AI 智慧會議時間媒合系統

**版本**：1.0  
**最後更新**：2026-05-29

---

## 1. 系統概述

### 目的
讓會議發起人建立一個投票頁面，受邀者透過連結勾選自己可參加的時段，系統即時統計並自動推薦出席率最高的時段。

### 核心價值
- 零安裝：純網頁，手機電腦皆可使用
- 即時同步：任何人送出投票，所有人立刻看到最新結果
- 智慧推薦：自動計算最佳時段，不需人工彙整

---

## 2. 使用者角色

### 投票者（Voter）
- 不需登入，輸入姓名即可使用
- 可勾選多個可參加時段
- 可修改自己已投的時段（重新勾選後送出即覆蓋）
- **無法**刪除他人投票或修改行程設定

### 管理員（Admin）
- 需輸入帳號密碼（SHA-256 雜湊驗證）才能進入管理模式
- 擁有所有投票者的權限
- 額外可執行管理操作（見第 4 節）
- 登出方式：重新整理頁面或關閉分頁

---

## 3. 功能規格

### 3.1 投票功能

| 項目 | 說明 |
|------|------|
| 身份輸入 | 文字欄位輸入姓名，或點選受邀成員名單快速填入 |
| 時段選擇 | 點選行事曆方格 Toggle 選取/取消，未登入不可操作 |
| 送出投票 | 按「送出投票」儲存，同一人重複送出視為更新 |
| 新成員自動加入 | 若輸入的姓名不在受邀名單，自動加入並計入未投票統計 |
| 已投票預載 | 輸入曾投過票的姓名，自動載入上次選擇供修改 |

### 3.2 即時同步

- 使用 Firebase Realtime Database WebSocket 連線
- 任何人送出投票，所有開著頁面的使用者在 1 秒內看到更新
- Header 顯示綠色「即時同步」脈衝指示器
- 斷線後 Firebase SDK 自動重連

### 3.3 智慧推薦（Agent 分析）

- 計算所有時段的出席人數
- 取出席人數最高的時段作為「最佳推薦時段」
- 若有並列最高則同時顯示所有並列時段
- 列出可參加最佳時段的人員名單
- 若仍有未投票者，顯示提醒文字

### 3.4 時段統計排行榜

- 列出所有日期時段，依出席人數由高至低排序
- 每個時段顯示：出席人數、佔受邀人數比率、進度條、可參加人員名單
- 最高出席時段以金色標示（第一名）

### 3.5 催票看板

- 已投票：顯示姓名 + 該人已勾選時段數
- 未投票：顯示尚未填寫的受邀成員
- 參與率：百分比即時計算
- 一鍵複製催票訊息（含未投票名單）

### 3.6 管理模式

進入條件：點「進入管理與設定模式」→ 輸入正確帳號密碼

| 功能 | 說明 |
|------|------|
| 修改會議標題 | 直接在 Header 編輯，退出管理模式時儲存 |
| 修改會議說明 | 同上 |
| 新增日期 | 使用日期選擇器，自動格式化含星期（如 2026-06-12 (五)） |
| 刪除日期 | 同步刪除該日期所有時段及相關投票紀錄 |
| 新增時段 | 選擇目標日期 + 輸入時段區間（如 14:00 - 16:00） |
| 刪除時段 | Hover 時段後顯示刪除按鈕，同步清除相關投票 |
| 邀請成員 | 新增成員至受邀名單，影響未投票統計 |
| 移除成員 | 同步刪除該成員的投票紀錄 |
| 刪除個別投票 | Hover 已投票成員名牌，顯示刪除按鈕（管理模式限定） |
| 清空所有投票 | 一鍵重設，操作前有二次確認 |

## 4. 資料模型

### Firebase Realtime Database 結構

```
meeting-v1/
├── title         String    會議標題
├── desc          String    會議說明
├── members       Array     受邀成員名單 ["小明", "阿華", ...]
├── slots         Array     可選時段
│   └── [i]
│       ├── id    String    "slot-{timestamp}"
│       ├── date  String    "2026-06-08 (一)"
│       └── times Array     ["09:30 - 11:30", "14:00 - 16:00"]
└── votes         Object    投票紀錄
    └── {姓名}    Array     ["2026-06-08 (一)_09:30 - 11:30", ...]
```

### 投票 key 格式

```
{date}_{time}
範例："2026-06-08 (一)_09:30 - 11:30"
```

### 日期格式

```
YYYY-MM-DD (週幾)
範例："2026-06-12 (五)"
排序方式：字典序 = 時間序（因 YYYY-MM-DD 前綴）
```

## 6. 安全性設計

### 管理員驗證

- 密碼以 SHA-256 雜湊儲存於前端原始碼（非明文）
- 驗證使用瀏覽器內建 `crypto.subtle.digest` API，不依賴第三方
- 帳號：`admin`
- 原始碼中僅存在雜湊值，無法直接反推密碼

### 資料庫規則

Firebase Realtime Database 建議規則：

```json
{
  "rules": {
    "meeting-v1": {
      ".read": true,
      ".write": true
    }
  }
}
```

> ⚠️ 現行規則為公開讀寫，任何人知道資料庫 URL 皆可透過 API 直接操作資料。管理員密碼保護僅為 UI 層級。

### 已知安全限制

| 限制 | 說明 |
|------|------|
| 無伺服器端身份驗證 | 投票者身份靠姓名區分，任何人可輸入他人名字覆蓋投票 |
| 管理員密碼可逆 | SHA-256 雜湊可被暴力破解，建議使用複雜密碼 |
| API 直接存取 | 知道 Firebase URL 者可繞過 UI 直接修改資料 |

## 7. 部署流程

### 前置條件

1. Firebase 專案已建立且 Realtime Database 啟用
2. `index.html` 中 `firebaseConfig` 已填入正確值
3. GitHub Pages 已啟用（Settings → Pages → main branch）

### 部署指令

```bash
git add .
git commit -m "deploy"
git push
```

GitHub Actions 或 Pages 自動部署，約 1–3 分鐘生效。

### 分享連結格式

```
https://{GitHub帳號}.github.io/{repo名稱}/
```
